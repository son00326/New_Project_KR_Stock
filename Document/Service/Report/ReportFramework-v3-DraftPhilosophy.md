# ReportFramework v3 — Design Philosophy (2-Layer Framework)

> **[v3.0 확정 전 중간 산출물. v3.0 확정 시 본 문서는 아카이브 예정]**

> **이 문서의 역할**: v3.0 전면 개정의 **설계 철학**. "체크리스트 프레임워크 → 독자 서사 프레임워크" 전환의 원칙을 박제.
> Narrative Design (`ReportFramework-v3-NarrativeDesign.md`)은 이 철학의 **구현 규격**이며, 이 문서는 **Why·When·충돌 시 우선순위**를 정한다.

---

## 1. 왜 v3인가 — v2.1 핵심 실패 진단

### 1.1 사용자 피드백 (2026-04-14)
> "너 사용자가 초보야… 내 산업군이 예를들어 건설인데, 바이오 내용 저렇게 두서없이 말하면 너 알아들어?"

### 1.2 실패 사례 진단 (`Document/Service/Report/ReportFramework-v3-ValuationTrial.md`)
250줄 §4 시범본에서 확인된 3대 실패 패턴:

| 패턴 | 사례 | 초보가 겪는 결과 |
|---|---|---|
| **메타 용어 노출** | "Peer 5축 필터링을 적용하면…", "Pure-play vs Secondary Comp 분리" | 보고서가 자기 분석 방법을 자랑하는 문서로 읽힘 |
| **용어 선행 설명 부재** | "LTM PSR 130.6배"를 문맥 없이 제시 | 첫 등장에서 이해 포기 |
| **표·매트릭스 과부하** | 5열 이상 비교표 4개 + Appendix 4-A~E 5개 | 어느 숫자를 봐야 하는지 결정 불가 |

### 1.3 근본 원인
v2.1은 **"작성자가 빠뜨리지 않기 위한 체크리스트"** 중심 설계.
→ 규격은 탄탄하나 "독자가 순차 이해하는 경로"가 없음.
→ 모든 섹션이 **병렬 나열**이지 **서사 진행**이 아님.

## 2. v3 핵심 전환 — 2-Layer Framework

### 2.1 두 레이어 정의

| Layer | 정체성 | 대상 | 소재 |
|---|---|---|---|
| **WHAT Layer (체크리스트)** | "무엇을 넣어야 하는가" | 작성자 | 10기법·Core Committee·Red Flag·Sources 6필드·P1~P8 등 |
| **HOW Layer (서사 가이드)** | "독자가 어떻게 따라오게 할 것인가" | 독자 | 5질문·비유 카드·오해 박스·3문장 요약·Transition |

### 2.2 두 레이어 관계

```
WHAT Layer   →  포함되어야 할 정보 집합 {a, b, c, ...}
HOW Layer    →  그 정보를 제시하는 순서·비유·박스 구조
최종 보고서   =  WHAT Layer 요소 전체를 HOW Layer 규격으로 표현
```

**작성 순서** (강제):
1. WHAT 수집 (document-specialist + analyst)
2. 5질문 매핑 (analyst)
3. 비유 카드 검색 (document-specialist ReaderAnalogyCards 참조)
4. HOW 구조로 조립 (writer)
5. 2-Layer 정합성 검증 (critic)

### 2.3 충돌 시 우선순위

| 충돌 상황 | 우선 | 이유 |
|---|---|---|
| WHAT 분량 초과 vs HOW 가독성 | **HOW 우선** | 독자가 못 읽으면 WHAT 완결 무의미 |
| 팩트 정확성 vs 비유 친화성 | **WHAT 우선** (팩트 훼손 금지) | 비유는 근사. 수치 오류는 치명 |
| Core Committee 전문 용어 vs 초보 가독성 | **비유 + 원문 병기** | "해자(경쟁사가 못 따라오는 벽, Moat)" |
| P7 Sources 6필드 vs 본문 미관 | **Appendix로** | 본문은 prose, 6필드는 Appendix 4-E |
| Red Flag Tier 1 자동 SELL vs 5질문 서사 | **Tier 1 우선** | 배너 최상단 + 서사 ①~④ 참고 강등 |

### 3.1 P8 기법들의 HOW 변환 (writer에게 가장 중요)

| P8 기법 (WHAT) | HOW 표현 방식 |
|---|---|
| ① Peer 5축 필터링 | "비교할 수 있는 회사는 딱 1곳입니다. 왜냐하면…" prose 1문단 |
| ② Pure-play vs Secondary | "진짜 같은 사업 vs 참고용 회사" 2그룹 표 (3열) |
| ④ LTM + NTM + 3Y | "최근·내년·3년뒤" 열 이름 한글화 |
| ⑦ 프리미엄 근거 | "비싼 이유 3가지 / 싼 이유 3가지" 박스 |
| ⑧ 사업모델 규칙 (R1~R3) | 비유 1문단 (로열티 회사 = 건설 공법 라이선스 모델) |
| Bridge 1 (75%ile) | "상위 25% 수준" 자연어 |
| ③ 통계 요약 (SHOULD) | "같은 업종 회사들 평균과 비교" 1문장 |
| ⑥ Moat (SHOULD) | "경쟁사가 못 따라오는 벽" 비유 |
| Bridge 2 (SHOULD) | "이 회사만의 강점이 확인되면 목표가 +10%" 조건 서술 |

## 4. 금지 원칙 (v2.1에 없던 새 규칙)

### 4.1 본문 금지 표현
- 분석 방법론 용어: `Peer 5축`, `Pure-play`, `Bridge 1/2/3`, `재분류`, `통계 요약`
- 영어 약자 무해설: `PSR`, `PER`, `WACC`, `DCF`, `LTM`, `NTM`, `OPM`, `EV/EBITDA`, `EWMA`, `CAGR`, `TP` (모두 첫 등장 시 한글 비유 후 병기)
- 영문 고유명사 무설명: `Moat`, `Bull/Base/Bear`, `Kelly`, `Position Size` (한글 풀이 후 병기)

### 4.2 본문 표 제약
- 4열 이상 금지 (Appendix로 이동)
- 섹션당 표 ≤2개
- 연속 표 2개 금지 (사이 prose 필수)

### 4.3 숫자 밀도
- 단락당 숫자 ≤3개
- 단락당 영어 약자 ≤2개

### 4.4 구조 금지
- 섹션 오프닝에 갑자기 표 등장 금지 (반드시 1문장 prose 선행)
- Summary 맨 위에 표 금지 (3문장 요약이 먼저)

## 12. v3.1 정정 (2026-04-14 세션 사용자 피드백 반영)

### 12.1 독자 페르소나 재정의
- **기존(v3.0 초안)**: 건설업 15년 차 특화
- **정정(v3.1)**: **"투자 초보 일반인"** — 직업 무관, 주식·바이오 지식 제로, 본인 자금 수천만~1억 관심 투자자
- 건설업 비유는 **예시 하나**였을 뿐, 모든 독자를 건설업으로 고정하면 IT·유통·서비스업 독자 몰입 실패

### 12.2 비유 전략 재설계
- **1순위 (본문)**: **범용 일상 비유** — 누구나 아는 것 (택시·월세·보험·편의점·유튜브 구독·카페 로열티 카드·프랜차이즈·부동산 임대·전기세·넷플릭스 등)
- **2순위 (보조)**: 섹터별 특화 비유 — "예를 들어 건설업이라면…" / "IT 업계라면…" 식으로 **병렬 제시** (1개 섹션에 최대 2개 업종 예시)
- **3순위 (부록)**: `ReaderAnalogyCards-ConstructionToBio.md` 같은 섹터별 카드첩은 **용어 사전 자산**으로 격하. 본문 의존 금지
- 용어 첫 등장 시 **일상 비유로 먼저** 풀고, 필요 시 섹터 예시 괄호로 병기

### 12.3 §5~§8 필수 섹션 복원 (v3.0 초안이 §0~§4만 작성한 것은 오류)
**v3.1 보고서는 §0~§8 전부 작성 필수**. 이전 초안의 "범위 §0~§4" 제한은 폐기.

| 섹션 | 주 질문 | 필수 요소 |

| §6 시나리오 | ④ Bull/Base/Bear 확률·TP | 3 시나리오 prose + 확률가중 TP 도출 + Pre-Mortem |

### 12.4 §8 투심위 규격 (v2 §8 그대로 유지, 가장 중요)
v2 알테오젠 §8(560~895줄)의 구조를 그대로 사용:
- **Part A. Sector Board 10명 독립 의견** — 국내 5 + 해외 5 (섹터별 슬롯 구성, 한 줄 의견 + 투표)
